linkhop — Matching-Logik (Abschnitt 2/5)

Wie aus einem Input-Link Treffer auf anderen Diensten werden

1. Normalisieren

URL parsen → source, type (track/album/artist), id. Unbekannt? → 400 mit klarer Fehlermeldung.

↓

2. Cache-Check

Key source:type:id in Redis. Hit → sofort zurück. Miss → weiter.

↓

3. Source-Resolve

Adapter des Input-Dienstes holt Metadaten: Titel, Artists, Dauer, Release-Datum, Artwork, isrc (Tracks) oder upc (Alben).

↓ für jeden Target-Dienst parallel ↓

3a. ID-Match

Tracks: Suche per ISRC. Alben: per UPC. Treffer → Confidence 1.0.

3b. Metadaten-Match

Fällt 3a aus → Suche per artist + title (+ duration ±2s). Top-3 Treffer scoren.

3c. Kein Treffer

Ergebnis markiert not_found. UI zeigt "nicht verfügbar", kein Fake-Link.

↓

4. Scoring

Confidence 0–1 aus: ID-Match (+1), Titel-Levenshtein, Artist-Overlap, Dauer-Abweichung. Schwelle ≥ 0.7 = "sicher"; 0.4–0.7 = "wahrscheinlich" (UI-Badge); <0.4 = not_found.

↓

5. Cache-Write + Response

Komplettes Resultat {source_meta, targets:[…]} im Redis mit TTL (Default 7 Tage). Ausgeliefert an Client.
